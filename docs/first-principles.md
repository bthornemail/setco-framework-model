# SETCO From First Principles

> "SETCO classifies. OMI defines. Projection is not authority."

## Root Problem

Modern computation mixes incompatible authority surfaces. A test is treated like a proof. A diagram is treated like an architecture. A compiler is treated like a source of truth. A runtime artifact is treated like the object model. A metaphor is treated like canon.

SETCO exists to prevent that collapse.

---

## The Six Principles

### 1. Distinction Before Authority

Classification precedes authority. Before a system can decide what is true, executable, or useful, it must first distinguish what kind of thing it is handling. A claim must be classified before it is allowed to do work.

### 2. Boundary Before Promotion

A thing may not be promoted until its boundary is known. Every claim must declare what it supports, what it does not support, where it fails, and what authority it does not possess.

### 3. OMI Is the Object Model

The Omi Object Model is the pure object/address/relation model. It owns omi-notation, omi-lisp, omi-port, null rings, gauge rings, bounded envelopes, and observer surfaces. OMI is not the blackboard, the canvas, or the OS. Those are surfaces that may adapt OMI — they do not replace it.

### 4. SETCO Is the Transition Framework

SETCO is the independent framework that allows existing computational environments to understand, classify, stage, and loosely connect to the Omi Object Model. SETCO owns methodology. OMI owns the object model.

### 5. Proof, Implementation, and Projection Must Remain Separate

`omi-axioms` proves admissible OMI invariants. `omi-isa` witnesses executable OMI behavior. SETCO classifies how those rails relate. None of these replace each other.

### 6. SETCO Does Not Create Authority

SETCO does not decide truth by itself. It classifies what is allowed to count as what. Proof canonizes admissible structure. Validation determines accepted state. Attestation witnesses accepted transition. Projection displays accepted state. No layer may impersonate another.
